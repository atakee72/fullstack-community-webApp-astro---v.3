## Auth gate
- All admin pages auth-gate at the page level (frontmatter `getSession()` + redirect to `/login` if no session). `moderation.astro` additionally checks `session.user.role !== 'admin'` and renders the §09 "no access" state inline (see below) rather than redirecting — the page itself is reachable by any logged-in user, only the queue content is gated.
- API endpoints under `/api/admin/*` gate via `requireAdminSession()` from `src/lib/auth.ts` — returns a pre-shaped 401/403 `Response` if there's no session or the session lacks admin role. Used by every `/api/admin/announcements/*` and `/api/admin/moderation/*` endpoint. No `ADMIN_USER_IDS` fallback.
- **Banned-admin gap**: `requireAdminSession()` checks role but not `isBanned` — a banned account that still has `role: 'admin'` keeps moderation access. Documented in `src/components/auth/kiosk/CLAUDE.md`; candidate to close when touching admin APIs next (not fixed here).
- `/admin/moderation` and `/admin/announcements` both use **`AdminLayout`** (kiosk, plum accent) — both admin surfaces are fully migrated off the legacy `BaseLayout`.

### API contracts
- **`GET /api/admin/moderation`** — query params `reviewStatus` (`pending`/`approved`/`rejected`/`reviewed` — `reviewed` = `{ $in: ['approved','rejected'] }`, i.e. "never pending"), `contentType`, `decision`, `source` (`ai_moderation`/`user_report` — the ⚑ Gemeldet filter sends `source=user_report`), `authorId`, `sortBy`/`sortOrder`, `limit`/`offset`, and **`urgentFirst`** (bool — sorts `{ decision: -1, [sortBy]: sortDir }`; since `'urgent_review' > 'pending_review' > 'approved'` lexicographically, a descending sort on `decision` floats urgent items to the top of every page). Response includes `items` (each enriched with `authorStrikes`/`authorIsBanned` via a join against `users` on `authorId`), `pagination`, and `counts` (`pending`/`approved`/`approvedWithWarning`/`rejected`/`urgent`) — the stat row and toggle badges read `counts` directly, independent of the current filter/page.
- **`GET /api/admin/moderation/author-strikes?authorId=`** — Ban-Bremse ledger: returns `{ strikes, isBanned, history: [{ date, contentType, reason, title }] }` (max 10, newest-first from the user doc's `strikeHistory`, cross-referenced against `flaggedContent` for titles). Called by `AdmRejectModal` only when `item.authorStrikes >= 2` (i.e. this rejection would be the 3rd strike).
- **`POST /api/admin/moderation/review`** — single-item action (`approve`/`reject`/`approve_with_warning`). Returns `{ success, userBanned?, strikeCount? }`; `userBanned` gates the §08 ban toast in `ModerationApp.handleRejectConfirm`.
- **`POST /api/admin/moderation/bulk-review`** — same actions, batched (`flaggedContentIds[]`, capped at 50 both client- and server-side). Returns `{ success, results: [{ id, status, strikeCount? }], bansTriggered }`. `bansTriggered` drives the §05 bulk toast and is capped by transition, see below.

### The two novel guards
1. **Ban-Bremse** (`AdmRejectModal.svelte`) — when `item.authorStrikes >= 2`, the reject modal escalates: fetches the strike ledger via `author-strikes`, shows it inline, and requires an explicit danger-tinted checkbox ("Ja, {author} sperren…") in addition to the reason text before the CTA (`canSubmit`) enables. Purely client-side gating on top of a server that would ban regardless — the guard exists so a ban is never a side-effect the admin didn't consciously acknowledge ("Sperren passiert nie beiläufig").
2. **Folgen-Vorschau** (`AdmBulkRejectModal.svelte` + `computeBulkDeltas()` in `adminModeration.ts`) — before a bulk reject fires, the modal renders a per-item strike delta table (`from → to`, "N. Treffer in dieser Auswahl" for repeat authors within the same selection, "WIRD GESPERRT" badge on the item that crosses the threshold) and requires an ack checkbox counting *distinct accounts* that will be banned. `computeBulkDeltas` does sequential per-author summation in **selection order** (must match the `flaggedContentIds` POST order) mirroring exactly how `processReviewAction` processes the batch server-side — `ban: true` is set only on the first item per author that crosses 3, not on every item past the threshold.

**Bans-per-transition server fix** (`src/lib/reviewAction.ts`, `processReviewAction`): `userBanned`/`bansTriggered` count only the **not-banned → banned transition**, guarded by `userUpdate.isBanned !== true` before setting `isBanned: true`. Without that guard, rejecting multiple pending items from an *already-banned* author (e.g. a bulk batch with 3 items from one author who's already at 3+ strikes) would re-set `userBanned = true` on every item, inflating `bansTriggered` in `bulk-review.ts` past the number of accounts actually crossing the line — verified with a single-author/3-item bulk fixture returning `bansTriggered: 1` (not 2) and exactly one `bannedAt` timestamp.

## Admin Official Announcements (`AnnounceApp.svelte` + `/admin/announcements.astro`)
**Kiosk design system** (`AdminLayout`, plum accent via `[data-page="admin"]` like moderation) — migrated from the legacy `BaseLayout` panel in July 2026 (Tasks 1–5). Page-level auth gate redirects (unlike moderation's in-page no-access state): logged-out → `/login?redirect=/admin/announcements`, non-admin → `/`.

**Page**: `src/pages/admin/announcements.astro` — SSR-fetches all officials (`{ isOfficial: true }`, sorted createdAt desc, limit 50) via `populateAuthors`, passes as `initialItems`. Mounts `<AnnounceApp client:only="svelte" initialItems adminName />` inside `AdminLayout` with `wordmark="amtliches"`, `backHref="/admin/moderation"`, `backLabel="← zur Moderation"`, `ribbonEcho="requireAdminSession()"`.

**Component roster** (`src/components/admin/kiosk/announce/`):
- `AnnounceApp.svelte` — orchestrator. Owns `items`/`status` state, the live kicker clock, `pinnedItem`/`archiveItems` derivations (`isPinned()` from `annFormat.ts` — server invariant: at most one item ever satisfies it), and every mutation (create, unpin, re-pin, edit, delete).
- `AnnCard.svelte` — board/archive card. Pinned-ness is derived internally from the item (`isPinned(item)`) rather than passed as a prop; `pending` prop overrides to the ink (pinned) treatment for an optimistically-created item before the server confirms. `compact` prop (driven by `AnnounceApp`'s own `matchMedia('(max-width: 767px)')` effect, same breakpoint as `AdminLayout`'s masthead split) tightens padding/font-size for the mobile stack.
- `AnnComposer.svelte` — dual-mode (`create`/`edit`) form, single instance reused across both via a `{#key}` block that remounts it whenever the compose target changes (entering/leaving edit, switching edit target, successful create) — this is what gives "clear on success" and "prefill on edit" without two-way binding into the parent. A failed submit does NOT change the key, so typed title/body survive (error state 04: "nichts verloren").
- `annFormat.ts` — pure formatting helpers (`isPinned`, `fmtPinDate`, `fmtCreated`, `fmtKickerDate`, `truncate`). No server imports — imported by the `client:only="svelte"` island (see root `CLAUDE.md` "Server-only modules bleeding into client bundles").

**Optimistic displacement + one-PATCH undo** (create flow, state 03): creating a new official optimistically inserts a `_pending: true` temp item at the top of the board AND (if something was already pinned) clears that item's `pinnedUntil` client-side — both wrapped in `withMove()` (see below) so the displaced card visibly animates from the board into the archive in the same frame the new card appears pinned. The real `POST /api/admin/announcements/create` follows; on success the temp item is swapped for the server-returned doc and a toast offers **undo** (`admin.ann.toast.undo`, 8s duration) *only if* the previous pin's original `pinnedUntil` hadn't already lapsed by the time undo would run. Undo is a single `PATCH { pinnedUntil: <original ISO> }` on the displaced item — not a client-side revert of the new post — followed by an animated `refetch()` so the server's post-undo state (which may re-clear the *new* item depending on displacement order) is authoritative. On a failed create, the optimistic state rolls back via `withMove()` and the composer shows error state 04 instead of losing input.

**`withMove()` — board↔archive View Transition**: a card moving between the board and archive sections crosses DOM subtrees (different `{#if}` branches), so a CSS transition class can't animate it — `AnnounceApp` wraps every state mutation that might move a card (create, unpin, re-pin, undo's animated refetch) in `document.startViewTransition()`, and every `AnnCard` wrapper carries a stable `view-transition-name="ann-<id>"` so the browser matches old/new position across the DOM swap. Falls back to a plain synchronous apply when `prefers-reduced-motion: reduce`, `document.startViewTransition` is unsupported, or SSR (`typeof document === 'undefined'`).

**`editCount` semantics**: incremented server-side on every successful `PATCH` to an announcement's title/body (edit, not pin/unpin toggles). `AnnCard` shows "{n}× bearbeitet" / "edited {n}×" in the meta line once `editCount >= 1`. Field lives on the `announcements` collection doc alongside the existing `isOfficial`/`pinnedUntil`.

**`pinnedUntil: null` vs expired chips**: `AnnCard` renders three distinct un-pinned states — `pinnedUntil: null` (never pinned or explicitly unpinned) shows the neutral "PIN GELÖST"/"UNPINNED" chip; a past-dated `pinnedUntil` (naturally expired, never explicitly cleared) shows "PIN ABGELAUFEN AM {date}"/"PIN EXPIRED {date}" instead — same dashed-border neutral styling, different copy, so an admin can tell "I unpinned this" from "this expired on its own" at a glance.

**Toast action bridge**: undo is wired through the shared `showToast(message, { type, action: { label, onClick }, duration })` bridge (`src/utils/toast.ts` → sonner) — same mechanism as other kiosk toasts, `action` just adds a clickable label inside the toast body.

**Mobile stack** (Task 5, `< md`): the desktop `460px 1fr` grid collapses to a single column. The composer column is always in the DOM (same `AnnComposer` instance, never remounted for responsive reasons) but is wrapped in a div that's `hidden` below `md` unless `AnnounceApp`'s `showMobileComposer` `$state` is true; a full-width ink CTA (`admin.ann.mobile.cta`, `min-height: 48px`, teal print-shadow) plus a centered mono hint (`admin.ann.mobile.hint`) sit above it and toggle that state — clicking always calls the same `focusComposer()` helper used by the empty-state CTA and by "✎ bearbeiten" (edit always force-opens + scrolls, per `handleEditClick` → `focusComposer()`). Section labels ("AM BRETT"/"ARCHIV" vs the desktop "AM BRETT — ANGEPINNT"/"ARCHIV — NICHT MEHR ANGEPINNT") are two parallel `md:hidden`/`hidden md:block` elements, not a JS string swap. Card footer actions get a `.ann-card-action` class (`src/styles/admin.css`) enforcing `min-height: 44px` below `md` only — desktop chip sizing is untouched.

- **Server invariant**: at most one official has `pinnedUntil > now` at a time. Enforced by atomic `updateMany` displacement on create (and on PATCH when pinnedUntil is being set to a future date) — see `/api/admin/announcements/create.ts` + `[id].ts`.
- **Bypass moderation**: admin officials skip the AI moderation pipeline entirely (admin is trusted). `moderationStatus: 'approved'` set directly. No flagged-content record created.
- **Pinning lifecycle**: 7-day default duration, replace-on-new (latest displaces previous), client-side check `pinnedUntil > now` at render time so no DB-side TTL is needed.
- **Known deviation from an early plan draft**: the create endpoint is `POST /api/admin/announcements/create` (not `/api/admin/announcements` with a bare POST) — the composer's mono echo line (`POST /api/admin/announcements/create`) documents this in the UI itself. Reviewed and upheld; don't "fix" it back.
