## Auth gate
- All admin pages auth-gate at the page level (frontmatter `getSession()` + redirect to `/login` if no session). `moderation.astro` additionally checks `session.user.role !== 'admin'` and renders the ¬ß09 "no access" state inline (see below) rather than redirecting ‚Äî the page itself is reachable by any logged-in user, only the queue content is gated.
- API endpoints under `/api/admin/*` gate via `requireAdminSession()` from `src/lib/auth.ts` ‚Äî returns a pre-shaped 401/403 `Response` if there's no session or the session lacks admin role. Used by every `/api/admin/announcements/*` and `/api/admin/moderation/*` endpoint. No `ADMIN_USER_IDS` fallback.
- **Banned-admin gap**: `requireAdminSession()` checks role but not `isBanned` ‚Äî a banned account that still has `role: 'admin'` keeps moderation access. Documented in `src/components/auth/kiosk/CLAUDE.md`; candidate to close when touching admin APIs next (not fixed here).
- `/admin/moderation` and `/admin/announcements` both use **`AdminLayout`** (kiosk, plum accent) ‚Äî both admin surfaces are fully migrated off the legacy `BaseLayout`.

### API contracts
- **`GET /api/admin/moderation`** ‚Äî query params `reviewStatus` (`pending`/`approved`/`rejected`/`reviewed` ‚Äî `reviewed` = `{ $in: ['approved','rejected'] }`, i.e. "never pending"), `contentType`, `decision`, `source` (`ai_moderation`/`user_report` ‚Äî the ‚öë Gemeldet filter sends `source=user_report`), `authorId`, `sortBy`/`sortOrder`, `limit`/`offset`, and **`urgentFirst`** (bool ‚Äî sorts `{ decision: -1, [sortBy]: sortDir }`; since `'urgent_review' > 'pending_review' > 'approved'` lexicographically, a descending sort on `decision` floats urgent items to the top of every page). Response includes `items` (each enriched with `authorStrikes`/`authorIsBanned` via a join against `users` on `authorId`), `pagination`, and `counts` (`pending`/`approved`/`approvedWithWarning`/`rejected`/`urgent`) ‚Äî the stat row and toggle badges read `counts` directly, independent of the current filter/page.
- **`GET /api/admin/moderation/author-strikes?authorId=`** ‚Äî Ban-Bremse ledger: returns `{ strikes, isBanned, history: [{ date, contentType, reason, title }] }` (max 10, newest-first from the user doc's `strikeHistory`, cross-referenced against `flaggedContent` for titles). Called by `AdmRejectModal` only when `item.authorStrikes >= 2` (i.e. this rejection would be the 3rd strike).
- **`POST /api/admin/moderation/review`** ‚Äî single-item action (`approve`/`reject`/`approve_with_warning`). Returns `{ success, userBanned?, strikeCount? }`; `userBanned` gates the ¬ß08 ban toast in `ModerationApp.handleRejectConfirm`.
- **`POST /api/admin/moderation/bulk-review`** ‚Äî same actions, batched (`flaggedContentIds[]`, capped at 50 both client- and server-side). Returns `{ success, results: [{ id, status, strikeCount? }], bansTriggered }`. `bansTriggered` drives the ¬ß05 bulk toast and is capped by transition, see below.

### The two novel guards
1. **Ban-Bremse** (`AdmRejectModal.svelte`) ‚Äî when `item.authorStrikes >= 2`, the reject modal escalates: fetches the strike ledger via `author-strikes`, shows it inline, and requires an explicit danger-tinted checkbox ("Ja, {author} sperren‚Ä¶") in addition to the reason text before the CTA (`canSubmit`) enables. Purely client-side gating on top of a server that would ban regardless ‚Äî the guard exists so a ban is never a side-effect the admin didn't consciously acknowledge ("Sperren passiert nie beil√§ufig").
2. **Folgen-Vorschau** (`AdmBulkRejectModal.svelte` + `computeBulkDeltas()` in `adminModeration.ts`) ‚Äî before a bulk reject fires, the modal renders a per-item strike delta table (`from ‚Üí to`, "N. Treffer in dieser Auswahl" for repeat authors within the same selection, "WIRD GESPERRT" badge on the item that crosses the threshold) and requires an ack checkbox counting *distinct accounts* that will be banned. `computeBulkDeltas` does sequential per-author summation in **selection order** (must match the `flaggedContentIds` POST order) mirroring exactly how `processReviewAction` processes the batch server-side ‚Äî `ban: true` is set only on the first item per author that crosses 3, not on every item past the threshold.

**Bans-per-transition server fix** (`src/lib/reviewAction.ts`, `processReviewAction`): `userBanned`/`bansTriggered` count only the **not-banned ‚Üí banned transition**, guarded by `userUpdate.isBanned !== true` before setting `isBanned: true`. Without that guard, rejecting multiple pending items from an *already-banned* author (e.g. a bulk batch with 3 items from one author who's already at 3+ strikes) would re-set `userBanned = true` on every item, inflating `bansTriggered` in `bulk-review.ts` past the number of accounts actually crossing the line ‚Äî verified with a single-author/3-item bulk fixture returning `bansTriggered: 1` (not 2) and exactly one `bannedAt` timestamp.

### Errors card (`AdmErrorsCard.svelte`) + Sentry proxy (`/api/admin/errors`)
**Placement**: `AdmErrorsCard` (no props) is mounted in `ModerationApp.svelte`'s desktop tree only, directly between `<AdmStatRow {counts} />` and `<AdmTitleBlock>` (both inside the same `hidden md:block` wrapper ‚Äî mobile stays triage-only, same precedent as Protokoll/bulk above).

**States** (five, driven by local `$state`, not TanStack): `loading` (skeleton bar) ‚Üí `unreachable` (502 from the proxy, or the `fetch` itself rejecting ‚Äî warn-tone text, no toast; the card is the signal) ‚Üí `disabled` (`{ enabled: false }`, i.e. `SENTRY_*` env vars unset ‚Äî the normal pre-setup state, not an error) ‚Üí `enabled === true && topIssues.length === 0` ("all clear" italic line) ‚Üí `enabled === true` with issues (total 24h count + up to 3 top-issue rows, each a link out to its Sentry `permalink`). A malformed/`error` JSON body is treated the same as `unreachable`.

**Refresh + guard**: self-fetches on mount, then every 5 minutes via `setInterval` (cleared `onDestroy`). A `fetchSeq` counter (mirrors `ModerationApp`'s own `fetchSeq` pattern) increments per request and every state write checks `seq === fetchSeq` first, so a slow in-flight response from a stale interval tick can't clobber a newer one.

**`GET /api/admin/errors`** (`src/pages/api/admin/errors.ts`) ‚Äî server-only proxy to the Sentry REST API so `SENTRY_AUTH_TOKEN` never reaches the client:
- **Guard runs BEFORE the env check**: `requireAdminSession(request)` is the very first thing the handler does ‚Äî an unauthenticated/non-admin caller gets the standard 401/403 and never learns whether Sentry is configured. Only after the guard passes does the handler read `SENTRY_AUTH_TOKEN`/`SENTRY_ORG`/`SENTRY_PROJECT` and return `{ enabled: false }` if any are missing.
- **Cache**: a 60-second module-level `{ data, expires }` cache, success-only (a 502 or thrown error is never cached, so a transient Sentry outage doesn't get pinned for a full minute). **Per-instance caveat**: Vercel can spin up multiple warm serverless containers, each with its own cache ‚Äî fine for a single-admin, infrequent-visit route; don't reach for Redis/Vercel KV unless that stops being true.
- **Timeout**: `AbortSignal.timeout(5000)` on the outbound `fetch` to `sentry.io/api/0/projects/{org}/{project}/issues/`.
- **Response contract**: `{ enabled: false }` (not configured) | `{ enabled: true, totalLast24h, topIssues: [{ id, title, permalink, count24h }] }` | 502 `{ error: 'sentry_unreachable' }`. `count24h` and `totalLast24h` are summed from each issue's `stats['24h']` bucket array, NOT Sentry's `count`/`userCount` fields (those are lifetime totals ‚Äî see root plan Decision 10). No `affectedUsers` field ‚Äî there is no honest 24h-scoped users source in this endpoint, so it's omitted rather than shown wrong.

**i18n**: 8 `admin.errors.*` keys (`kicker`, `total`, `top`, `none`, `open`, `disabled`, `unreachable`, `count`) in `src/lib/kiosk-i18n.ts`, DE + EN, both real (unlike the modal DE-contract keys above ‚Äî this card's copy is properly translated both ways).

## Admin Official Announcements (`AnnounceApp.svelte` + `/admin/announcements.astro`)
**Kiosk design system** (`AdminLayout`, plum accent via `[data-page="admin"]` like moderation) ‚Äî migrated from the legacy `BaseLayout` panel in July 2026 (Tasks 1‚Äì5). Page-level auth gate redirects (unlike moderation's in-page no-access state): logged-out ‚Üí `/login?redirect=/admin/announcements`, non-admin ‚Üí `/`.

**Page**: `src/pages/admin/announcements.astro` ‚Äî SSR-fetches all officials (`{ isOfficial: true }`, sorted createdAt desc, limit 50) via `populateAuthors`, passes as `initialItems`. **Component roster** (`src/components/admin/kiosk/announce/`):
- `AnnounceApp.svelte` ‚Äî orchestrator. Owns `items`/`status` state, the live kicker clock, `pinnedItem`/`archiveItems` derivations (`isPinned()` from `annFormat.ts` ‚Äî server invariant: at most one item ever satisfies it), and every mutation (create, unpin, re-pin, edit, delete).
- `AnnCard.svelte` ‚Äî board/archive card. Pinned-ness is derived internally from the item (`isPinned(item)`) rather than passed as a prop; `pending` prop overrides to the ink (pinned) treatment for an optimistically-created item before the server confirms. `compact` prop (driven by `AnnounceApp`'s own `matchMedia('(max-width: 767px)')` effect, same breakpoint as `AdminLayout`'s masthead split) tightens padding/font-size for the mobile stack.
- `AnnComposer.svelte` ‚Äî dual-mode (`create`/`edit`) form, single instance reused across both via a `{#key}` block that remounts it whenever the compose target changes (entering/leaving edit, switching edit target, successful create) ‚Äî this is what gives "clear on success" and "prefill on edit" without two-way binding into the parent. A failed submit does NOT change the key, so typed title/body survive (error state 04: "nichts verloren").
- `annFormat.ts` ‚Äî pure formatting helpers (`isPinned`, `fmtPinDate`, `fmtCreated`, `fmtKickerDate`, `truncate`). No server imports ‚Äî imported by the `client:only="svelte"` island (see root `CLAUDE.md` "Server-only modules bleeding into client bundles").

**Optimistic displacement + one-PATCH undo** (create flow, state 03): creating a new official optimistically inserts a `_pending: true` temp item at the top of the board AND (if something was already pinned) clears that item's `pinnedUntil` client-side ‚Äî both wrapped in `withMove()` (see below) so the displaced card visibly animates from the board into the archive in the same frame the new card appears pinned. The real `POST /api/admin/announcements/create` follows; on success the temp item is swapped for the server-returned doc and a toast offers **undo** (`admin.ann.toast.undo`, 8s duration) *only if* the previous pin's original `pinnedUntil` hadn't already lapsed by the time undo would run. Undo is a single `PATCH { pinnedUntil: <original ISO> }` on the displaced item ‚Äî not a client-side revert of the new post ‚Äî followed by an animated `refetch()` so the server's post-undo state (which may re-clear the *new* item depending on displacement order) is authoritative. On a failed create, the optimistic state rolls back via `withMove()` and the composer shows error state 04 instead of losing input.

**`withMove()` ‚Äî board‚Üîarchive View Transition**: a card moving between the board and archive sections crosses DOM subtrees (different `{#if}` branches), so a CSS transition class can't animate it ‚Äî `AnnounceApp` wraps every state mutation that might move a card (create, unpin, re-pin, undo's animated refetch) in `document.startViewTransition()`, and every `AnnCard` wrapper carries a stable `view-transition-name="ann-<id>"` so the browser matches old/new position across the DOM swap. Falls back to a plain synchronous apply when `prefers-reduced-motion: reduce`, `document.startViewTransition` is unsupported, or SSR (`typeof document === 'undefined'`).

**`editCount` semantics**: incremented server-side on every successful `PATCH` to an announcement's title/body (edit, not pin/unpin toggles). `AnnCard` shows "{n}√ó bearbeitet" / "edited {n}√ó" in the meta line once `editCount >= 1`. Field lives on the `announcements` collection doc alongside the existing `isOfficial`/`pinnedUntil`.

**`pinnedUntil: null` vs expired chips**: `AnnCard` renders three distinct un-pinned states ‚Äî `pinnedUntil: null` (never pinned or explicitly unpinned) shows the neutral "PIN GEL√ñST"/"UNPINNED" chip; a past-dated `pinnedUntil` (naturally expired, never explicitly cleared) shows "PIN ABGELAUFEN AM {date}"/"PIN EXPIRED {date}" instead ‚Äî same dashed-border neutral styling, different copy, so an admin can tell "I unpinned this" from "this expired on its own" at a glance.

**Toast action bridge**: undo is wired through the shared `showToast(message, { type, action: { label, onClick }, duration })` bridge (`src/utils/toast.ts` ‚Üí sonner) ‚Äî same mechanism as other kiosk toasts, `action` just adds a clickable label inside the toast body.

**Mobile stack** (Task 5, `< md`): the desktop `460px 1fr` grid collapses to a single column. The composer column is always in the DOM (same `AnnComposer` instance, never remounted for responsive reasons) but is wrapped in a div that's `hidden` below `md` unless `AnnounceApp`'s `showMobileComposer` `$state` is true; a full-width ink CTA (`admin.ann.mobile.cta`, `min-height: 48px`, teal print-shadow) plus a centered mono hint (`admin.ann.mobile.hint`) sit above it and toggle that state ‚Äî clicking always calls the same `focusComposer()` helper used by the empty-state CTA and by "‚úé bearbeiten" (edit always force-opens + scrolls, per `handleEditClick` ‚Üí `focusComposer()`). Section labels ("AM BRETT"/"ARCHIV" vs the desktop "AM BRETT ‚Äî ANGEPINNT"/"ARCHIV ‚Äî NICHT MEHR ANGEPINNT") are two parallel `md:hidden`/`hidden md:block` elements, not a JS string swap. Card footer actions get a `.ann-card-action` class (`src/styles/admin.css`) enforcing `min-height: 44px` below `md` only ‚Äî desktop chip sizing is untouched.

- **Server invariant**: at most one official has `pinnedUntil > now` at a time. Enforced by atomic `updateMany` displacement on create (and on PATCH when pinnedUntil is being set to a future date) ‚Äî see `/api/admin/announcements/create.ts` + `[id].ts`.

- **Pinning lifecycle**: 7-day default duration, replace-on-new (latest displaces previous), client-side check `pinnedUntil > now` at render time so no DB-side TTL is needed.
